## 1. 两篇文章核心观点与方法论（分析）

### 文章 A《一个人、6个Agent、月入五万：搭建一人AI公司架构》（空空，2026-05-27）
- **核心论点**：一人 AI 公司的壁垒是"编排"而非 AI 本身；6 个 Agent（Scout 选题侦察 / Strategist 选题决策 / Writer 写手 / Visual 配图 / Publisher 发布 / Analyst 复盘）形成闭环。
- **验证节奏（五周跑一轮）**：1–2 周手搓脚本闭环验证单场景（不依赖框架）；3–4 周连续运营 14 天对比人工（阅读量/互动率/写作时间/单篇成本）；第 5 周决策——AI 数据≥人工 70% 且成本≤1/5 才值得继续。
- **五个坑**：① 平台政策（定位"AI 辅助创作+自动化运营"，非"AI 帮你写"）② 同质化（护城河=数据飞轮/长期记忆，非代码）③ 内容质量（限频+质量门槛，过不了 AI 检测不让发）④ 法律风险（配图走授权库/用户自上传）⑤ 用户预期（**绝不说"零人工"**，定位"AI 起草，你审核"）。
- **最关键一环**：Analyst 把数据写回 Strategist 长期记忆 → 越用越懂用户的飞轮；没有它，系统只是各自为战的工具。

### 文章 B《一人公司OPC如何用公众号＋智能体，验证AI产品第一步？》（陈宇明，2026-07-13）
- **核心论点**：MVP 阶段先别做重；"判断需求值不值得做"比"把产品做出来"更稀缺。
- **微信生态六步法**：① 微信指数查**用户真实会搜的词**（看趋势是否稳定 + 搜一搜占比是否高）② 关键词对比（加对比词，选长期稳定高于且搜一搜占比高的词）③ 判断值不值得（指数高+趋势稳+搜一搜占比高）④ 注册公众号（订阅号，**名称用用户搜索词**，不取抽象品牌名；AI 生成 logo+SEO 介绍）⑤ 用**智能体跑最小体验**（只需提示词，先不开发小程序）⑥ 打通公众号+智能体 + **观察转化数据**（不看阅读量，看点击智能体/发起对话/连续追问/关注/收藏分享）决定下一步。
- **理念**：不要先创造概念，先承接需求；AI 时代 MVP 可以是一个智能体；公众号种草承接信任，智能体解决问题验证需求。

## 3. 增量优化清单（4 项，含来源 + 理由 + 触发/输入/输出/场景）

### 优化 ①【来源：文章 B 全篇 + 文章 A"先验证再开发"】【落到：opc-mvp-designer】
**优化理由**：本阶段原只定"验证什么/最小形式/成功标准/边界/人机分工"，缺具体轻量验证手段。文章 B 提供可直接落地的微信生态六步法，把 MVP 验证成本降到极低，契合精益验证理念；文章 A 补入"先验证再开发、先跑通一个号再规模化"节奏。
**新增小节**：`## 轻量需求验证协议（微信生态六步法）`
- 触发条件：用户在微信生态做一人公司且验证形式选"内容验证/工具原型"，或对"需求是否真实"存疑需先数据证伪。
- 输入：产品想法一句话 + `04-business-model/risky-assumptions.md`。
- 六步（仅搭验证闭环，不写正文）：微信指数查词 → 关键词对比 → 判断值不值得 → 注册公众号(订阅号,搜索词命名) → 智能体跑最小体验 → 打通+观察转化数据。
- 输出：被验证的切入关键词 + 轻量验证闭环(公众号+智能体) + "继续开发/放弃"的数据决策门槛（AI≥人工70%且成本≤1/5）。
- 边界：只用于确定"最小验证形式"，**不进入帖子/文案撰写**（属执行期或 `opc-asset-ops`）。

### 优化 ②【来源：文章 A"Analyst→记忆飞轮"】【落到：opc-dashboard-review】
**优化理由**：本技能原只做"找瓶颈→定下一重点"并落盘，缺"复盘结论持续写回记忆/案例库"这一步。文章 A 强调这是最关键的一环——没有它系统是各自为战的工具，有它才越用越聪明。
**新增小节**：`## 数据飞轮写回（复盘→记忆/案例库）`
- 触发条件：每次复盘确认瓶颈与下一重点后，立即执行。
- 输入：本次 `review-[YYYYMMDD].md` + 历史 `dashboard.json`。
- 写回动作（按可用度选做）：① 假设置信度→更新 `state/assumptions.json`；② 偏好/模式记忆→写回 `.workbuddy/memory/` 或 `08-asset-ops/` 案例库/模式库；③ 暴露可复用成果→提示触发 `opc-asset-ops` 并在 `asset-register` 追加。
- 输出：写回动作已执行（或标注"本次无需写回"），与 review 一并留存。
- 衔接：写回的记忆会被下次 `opc-niche-positioning`(环3)/`opc-mvp-designer`(验证形式)/`opc-asset-ops`(资产优先级) 自动读取，形成闭环。

### 优化 ③【来源：文章 A 坑2(护城河=数据飞轮) + 坑1/坑5(对外话术定位)】【落到：opc-asset-ops】
**优化理由**：本阶段原做资产优先级但未突出"带记忆/数据飞轮属性的资产应优先"；合规护栏缺"对外话术定位"。两文共同指向——一人公司壁垒是数据飞轮与信任，不是代码或功能堆砌。
**新增小节**：`## 护城河资产优先级原则 + 合规对外话术定位`
- 护城河优先级：优先沉淀案例库/用户偏好记忆/爆款模式库/复盘结论库/选题模式库（用越久越懂用户）；次优先一次性内容模板/SOP；纯代码若不积累数据则护城河弱。
- 合规对外话术红线（写入 C2）：定位「AI 辅助创作+自动化运营」「AI 起草你审核」，**绝不**宣传"零人工/全自动代写"；平台侧按规则定位规避识别；配图走授权库或用户自上传。
- 衔接：与优化②互为因果——复盘写回的记忆，正是此处优先沉淀的护城河资产。

### 优化 ④【来源：文章 B 第四步(命名用用户搜索词)】【落到：opc-niche-positioning】
**优化理由**：本阶段只做"是谁/痛什么/为什么有切口"，不延伸到执行；但"切口"能否被搜到高度依赖命名是否命中用户已有搜索习惯。补一条轻量可选命名原则，作为利基确认后的战术提示（不写文案、不越界）。
**新增小节**：`## 命名与入口承接原则（可选战术）`
- 原则（仅提示，不强制、不写正文）：公众号/智能体/店铺名称优先用"用户真实会搜的词"，一眼知干嘛；不取抽象/故事感/需被教育的品牌名；核心理念"先承接已有需求，再谈教育用户"。
- 衔接：命名确定后若走微信生态验证，直接对接优化①的第 4 步（注册公众号）。

## 4. 衔接与去重说明

- **无新增平行 skill**：4 项分别落在 02/06/08/09 四个不同且不重叠的阶段，沿用现有 skill 的"边界/越界检测/落盘检查点"机制，天然无冲突。
- **飞轮闭环**：优化②（09 写回）↔ 优化③（08 护城河资产）互为因果；优化②写回的记忆被优化①（06 验证形式选择）、优化④（02 命名）、优化③（08 优先级）自动读取。
- **边界守护**：优化①/④ 都明确"只搭验证闭环/只给命名原则，不写文案"，不越入执行层，与现有各 skill 的"越界检测"一致。
- **与既有合规一致**：优化③的对外话术红线与项目 D-08(半自动)/D-09(不挂链)/D-11(不代写) 完全同向，仅补充"对外表达"维度。
