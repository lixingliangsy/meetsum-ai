## 1. 瓶颈诊断（铁三角：流量 / 转化 / 交付）

> 教学要点：复盘不是流水账，是找瓶颈。本周期看的是「需求 / 转化 / 资产 / 风险」四条线，目标不是证明自己努力了，而是找出**最值得修正的一点**——这一步会决定下一周期的唯一重点。

交付层确实没上线——Supabase、LLM、Waffo 产品 ID 三类凭证全是 REPLACE_ME，deploy.sh 门禁会拦死真部署，产品现在对外部用户不可用。但这一层的卡点**不在我们手里**：把它挂载到 06 的 **H0 上线三缺口（P0 前置闸门）**——它是用户一次性填凭证 + 执行迁移 SQL + 部署就能解的"外部阻塞"，属于临时、一次性的被动等待，不是团队能持续发力的瓶颈。转化层同理——支付/计量没真跑，但把它挂载到 **H1 保真溢价 / H5 订阅单位经济**，二者依附于 H0 先打通才能验，单独看没有独立瓶颈。

真正卡在团队可掌控范围里的是**流量层完全没启动**：把它挂载到 06 的 **H4 站外冷启动**——这是团队此刻就能拉、却空着没拉的杠杆；资产 8「案例库→小红书对比笔记草稿生成器」已经实跑验证过，却没排进日历、没产出真实内容。

**结论：两者叠加，但当前唯一可掌控、且错过就不可逆的瓶颈是「流量层未启动」（= H4 前置，同时也是 H0 上线闸门第四步）。**
- 交付层 = H0 上线三缺口（P0 前置闸门）：被动等用户填凭证，一次性、不在我方控制；
- 流量层 = H4 站外冷启动：我们此刻就能拉、却空着没拉的杠杆；
- 叠加风险：7 月毕业季是时间窗口，等凭证到了再起号，旺季流量红利就过去了。

所以现在不是"等产品上线"，而是"产品上线前先把内容和流量资产跑起来"。

## 2. 下周期单一重点（只给一条明确动作）

**上线前先把站外内容/流量资产跑起来（不依赖凭证），对齐 07 四路触达口径：用已验证的资产 8 生成器 + 07 路径 C（小红书种草标识 AI 不挂链）/ 路径 D（目录站工具导航）发出第一批"降重前后 AI 率对比"笔记与目录站提交，建起"产品上线即有受众"的预热队列；同时把 07 口径下的 H0 凭证采集动作——即已落地的 PRELAUNCH-CHECKLIST（凭证采集表）——作为消除 H0 外部阻塞的动作。**

具体动作：
- **流量侧（H4 前置 · 现在可跑）**：以资产 8 生成器的 `--sample` 模式本地批量产出 3–5 篇"降重前后 AI 率对比"笔记草稿，按 07 路径 C 口径（小红书种草标识 AI、不挂链、不 AI 托管发布 D-08）排期发出真实第一篇；并把 Product Hunt / HN / Indie Hackers / SaaSHub 等目录站/社区提交作为 07 路径 D 的轻量前置准备（提交待 GTM 阶段启用，但现在可备齐素材，呼应 06 §4.2 B1 重设渠道组合）。
- **交付侧（消除 H0 阻塞）**：把"用户持有的外部阻塞"转成一个 15 分钟动作——PRELAUNCH-CHECKLIST 凭证采集表，让用户一次性填齐 Supabase（URL/publishable/secret/DB_URL）/ LLM API Key / Waffo 产品 ID，跑通 deploy.sh 门禁即真部署解锁。

为什么不是另两个选项：
- 不是"等凭证上线"——那是被动等（H0 第一、二、三步），且会错过 7 月窗口；
- 不是"先收窄验证最小假设"——产品代码已全通（tsc 通过 M0.1→M1.2），降重/降AI 价值已被资产 8 的样例案例自证过，现在缺的不是假设验证，是流量入场（H4）。
